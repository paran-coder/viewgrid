# Stage 3 자체 점검 보고서

> 평가일: 2026-07-19  
> 버전: 0.3.0  
> 범위: 9개 실제 생성 큐, 실패·중단 재시도, 3×3 PNG, ZIP

## 1. 평가 요약

Stage 3의 핵심 목표인 멀티카메라 순차 생성과 브라우저 내 결과 패키징을 구현했다. 한 번의 원본 전처리 결과를 여러 카메라 요청에 재사용하며, 카메라별 상태와 전체 진행률을 분리해 표시한다.

완료 결과는 부분 실패나 사용자 중단 뒤에도 유지된다. 현재 카메라 설정과 생성 시점 설정이 다른 결과는 `STALE`로 표시하고, 최신 결과가 모두 갖춰지기 전에는 3×3 PNG를 허용하지 않는다.

## 4. 발견한 문제와 수정

### 4.1 큐 Mock 응답 본문 재사용

하나의 `Response` 인스턴스를 여러 요청에 반환하면 첫 번째 요청이 본문을 소비한 뒤 다음 요청에서 `Body has already been read`가 발생했다.

**수정:** Mock 공급자가 호출마다 새로운 `Response`를 생성하도록 변경하고 순차 큐 회귀 테스트를 추가했다.

### 4.2 카메라 변경 후 오래된 결과

이미지를 생성한 뒤 Yaw나 FOV를 바꿔도 이전 결과가 정상 완료처럼 표시됐다.

**수정:** 결과 메타데이터의 카메라 스냅샷과 현재 카메라를 비교한다. 값이 다르면 `STALE`로 표시하고 3×3 완료 수에서 제외한다.

### 4.3 큐 도중 API 키 삭제

탭 유지 옵션을 끈 경우 단일 요청 정책을 그대로 적용하면 첫 번째 셀 이후 키가 사라질 수 있었다.

**수정:** 키를 큐 시작 시 메모리 스냅샷으로 잡고, 전체 큐의 `finally`에서만 삭제한다.

### 4.4 실패 한 건이 전체 큐를 중단할 가능성

모든 공급자 오류를 동일하게 처리하면 일시적인 속도 제한 하나가 나머지 카메라를 모두 막을 수 있었다.

**수정:** `retryable` 정보를 클라이언트까지 유지한다. 재시도 가능한 오류는 해당 셀만 실패 처리하고 다음 셀을 실행하며, 인증·권한 같은 비재시도 오류만 남은 큐를 중단한다.

### 4.5 Blob 메모리 정리

재생성·삭제·새 프로젝트 과정에서 이전 Blob URL이 남을 수 있었다.

**수정:** 셀 교체 직전, 개별 삭제와 프로젝트 초기화에서 `URL.revokeObjectURL`을 실행한다.

## 5. 미완료 또는 환경 제한

### 실제 공급자 검증

사용자 API 키가 제공되지 않아 OpenAI와 Gemini의 실제 9개 생성 요청은 실행하지 않았다. 공급자 호출은 Route Handler와 Mock 통합 테스트로 검증했다.

### 브라우저 E2E

현재 실행 환경에는 Playwright가 요구하는 Chromium 바이너리가 설치되어 있지 않아 로컬 E2E를 완료하지 못했다. 이전 시스템 Chromium은 관리 정책 제약도 있어 사용하지 않았다. GitHub Actions는 Playwright 전용 Chromium을 설치해 다음을 검증하도록 구성했다.

- 단일 실제 생성
- 활성 카메라 9개 생성
- 결과 9/9 완료
- 3×3·ZIP 버튼 활성화
- ZIP 다운로드 이벤트
- axe 접근성 검사
- 360px·768px·1440px 레이아웃

### Next.js 전체 빌드

Next.js 16 프로덕션 빌드가 완료됐다. 컴파일, TypeScript, 페이지 데이터 수집, 정적 페이지 6개 생성과 최종 최적화까지 통과했다.

### 메모리 실측

2,328×2,328 Canvas의 이론적 RGBA 메모리는 약 21MB이며 생성 결과 Blob 9개가 추가된다. 실제 9개 고해상도 출력의 모바일 메모리 사용량은 Stage 6에서 측정해야 한다.

## 6. 의존성 점검

`npm audit`은 Next.js 내부의 PostCSS 경로에 중간 등급 2개를 보고했다. 자동 수정안이 현재 Next 16에서 Next 9로 내려가는 비정상적인 메이저 변경이어서 적용하지 않았다. 공식 패치 버전 확인과 업그레이드는 Stage 6 보안 점검에 포함한다.
