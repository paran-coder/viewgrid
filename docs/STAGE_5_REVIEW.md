# ViewGrid Stage 5 자체 점검 보고서

> 완료일: 2026-07-19 (Asia/Seoul)  
> 버전: 0.5.0  
> 단계: 멀티뷰 일관성 개선 및 정식 1.0 후보

## 1. 목표

Stage 5의 목표는 Stage 4에서 구현한 원본+로컬 가이드 생성 흐름을 여러 카메라에 일관되게 확장하는 것이었다.

핵심 목표:

1. 중앙 기준 결과를 먼저 생성한다.
2. 이후 카메라가 이미 완성된 인접 결과를 참조한다.
3. 공급자 결과의 크기와 전역 색조를 보수적으로 정규화한다.
4. 생성 계보와 설정 컨텍스트를 결과에 기록한다.
5. 카메라·가이드·일관성 설정을 재사용할 수 있게 한다.
6. API 키와 이미지를 영구 저장하지 않는다.

## 2. 완료된 기능

### 2.1 중앙 기준 생성 계획

기존의 단순 `C1 → C9` 처리를 다음 계획으로 교체했다.

```text
C5 → C4 → C6 → C2 → C8 → C1 → C3 → C7 → C9
```

- C5를 정면·중앙 기준으로 먼저 생성한다.
- C4·C6은 C5를 참조한다.
- C2·C8은 C5를 참조한다.
- C1·C7은 C4를 참조한다.
- C3·C9은 C6을 참조한다.
- 비활성 카메라는 계획에서 제외한다.
- 선택 재시도에도 같은 정책을 적용한다.

관련 구현:

- `src/lib/multiview-consistency.ts`
- `src/store/use-studio-store.ts`

### 2.2 인접 결과 참조

- 현재 카메라 설정과 일치하는 완료 결과만 참조 후보로 사용한다.
- 공급자, 모델, 가이드 모드, 참조 모드, 정규화 모드가 다른 결과는 제외한다.
- 참조 결과를 최대 1,024px·820KB WebP로 압축한다.
- 참조 파일 생성에 실패하거나 크기 제한을 넘으면 참조 없이 계속 생성한다.
- 생성 중 현재 사용할 참조 카메라를 UI에 표시한다.
- 결과 셀과 상세 메타데이터에 실제 참조 카메라를 기록한다.

### 2.3 공급자별 세 번째 이미지 입력

공통 입력 역할:

```text
Image 1: 원본
Image 2: 로컬 가이드(선택)
Image 3: 인접 결과(선택)
```

- OpenAI 어댑터는 순서가 있는 `image[]`로 전달한다.
- Gemini 어댑터는 텍스트 지시 뒤 원본·가이드·참조 순서로 구성한다.
- 프롬프트는 원본이 최우선이며 참조 결과와 충돌할 경우 원본을 따르도록 명시한다.
- Route Handler는 참조 MIME·크기와 전체 입력 합산 크기를 검증한다.

관련 구현:

- `src/lib/providers/openai.ts`
- `src/lib/providers/gemini.ts`
- `src/app/api/generate/route.ts`
- `src/lib/camera-prompt.ts`

### 2.4 출력 정규화

생성 결과는 옵션이 켜져 있을 때 브라우저에서 다음 처리를 거친다.

- 1,024×1,024 정사각형 캔버스
- 비율을 유지한 contain 배치
- WebP 재인코딩
- 원본 또는 실제 참조 결과의 샘플 평균색과 비교
- 제한된 RGB·휘도 게인 적용
- 픽셀 접근 실패 시 색상 보정 없이 크기만 정규화

제한 범위:

- RGB 채널 게인: 0.84–1.16
- 휘도 게인: 0.90–1.10

이는 피사체 마스크 또는 의미론적 배경 통일이 아니라, 결과 시트의 전역 크기·톤 차이를 줄이는 후처리다.

관련 구현:

- `src/lib/output-normalization.ts`

### 2.5 결과 컨텍스트와 STALE 판정

결과는 다음 생성 조건을 스냅샷으로 저장한다.

- 공급자
- 모델
- 카메라 파라미터
- 가이드 사용 여부
- 인접 결과 참조 사용 여부
- 출력 정규화 사용 여부

현재 설정과 하나라도 다르면 결과를 `STALE`로 표시하고 3×3 최신 완료 조건에서 제외한다.

결과 메타데이터에는 다음이 추가됐다.

- 참조 요청·실제 사용 여부
- 참조 카메라 ID
- 참조 파일 크기와 경고
- 생성 계획 버전
- 정규화 요청·실제 적용 여부
- 정규화 버전
- 출력 크기
- RGB 게인
- 정규화 기준 카메라와 경고

## 3. 자체 점검 중 발견한 문제와 수정

### 문제 1 — 생성 순서가 참조 계보와 맞지 않음

기존 C1부터의 순차 처리는 모서리·상하 결과가 중앙 기준 없이 먼저 생성될 수 있었다.

**수정:** 중앙 C5와 좌우 중앙 C4·C6을 먼저 처리하는 고정 계획을 추가했다.

### 문제 2 — 오래된 결과가 참조로 재사용될 위험

카메라 설정 또는 모델이 바뀐 결과도 완료 상태만 보고 참조할 수 있었다.

**수정:** 카메라와 전체 결과 컨텍스트가 현재 조건과 일치하는 경우만 참조한다.

### 문제 3 — 참조 이미지 추가로 Vercel 입력 한도에 가까워짐

원본+가이드에 참조까지 추가되면서 요청 크기 위험이 커졌다.

**수정:** 참조 전용 압축, 개별 상한, 전체 입력 합산 3.75MB 검증을 추가했다.

### 문제 4 — 참조 압축 실패가 전체 큐를 중단할 수 있음

참조는 보조 입력인데도 로컬 처리 실패가 유료 생성 전체를 막을 수 있었다.

**수정:** 참조 없이 원본+가이드로 계속 처리하고 경고를 메타데이터에 기록한다.

### 문제 5 — 공급자 또는 일관성 옵션이 다른 결과가 한 시트에 섞임

기존 STALE 판정은 카메라와 가이드 중심이었다.

**수정:** 공급자·모델·참조·정규화 모드까지 결과 컨텍스트에 포함했다.

### 문제 6 — 전역 색상 보정이 과도해질 가능성

단순 평균색 비율을 그대로 적용하면 제품색을 훼손할 수 있었다.

**수정:** 채널과 휘도 게인을 좁은 범위로 제한하고 픽셀 접근 실패 시 안전 폴백을 사용한다.

### 문제 7 — 세션 복구가 민감정보 저장으로 확대될 위험

상태 전체를 직렬화하면 API 키나 이미지 참조가 포함될 수 있었다.

**수정:** 전용 프리셋 스키마를 만들고 허용된 설정 필드만 저장한다.

### 문제 8 — 옵션 변경 후 결과가 최신처럼 표시됨

참조 또는 정규화 토글을 변경해도 기존 결과를 그대로 사용할 수 있었다.

**수정:** 두 옵션을 결과 컨텍스트와 STALE 판정에 추가했다.

## 5. 보안 및 개인정보 점검

- API 키는 메모리 상태에만 보관한다.
- API 키는 JSON 프리셋과 sessionStorage에 포함되지 않는다.
- 원본·가이드·참조·결과 이미지는 sessionStorage에 포함되지 않는다.
- Route Handler는 이미지 타입·개별 크기·합산 크기를 검증한다.
- 응답은 `Cache-Control: no-store`를 사용한다.
- 공급자 원문 오류와 API 키를 사용자 응답에 반사하지 않는다.
- ZIP Manifest에는 API 키와 Blob URL을 포함하지 않는다.

Stage 6에서 CSP, Permissions Policy, Referrer Policy, 속도 방어와 배포 환경 헤더를 최종 점검한다.

## 6. 알려진 한계

1. 생성 모델은 실제 3D 카메라 엔진이 아니므로 정확한 각도를 보장하지 않는다.
2. 참조 이미지가 일관성을 높일 수 있지만 참조 자체의 오류가 후속 결과에 전파될 수 있다.
3. 로고와 작은 문자는 큰 각도에서 여전히 변형될 수 있다.
4. 출력 정규화는 전역 평균 기반이며 피사체 마스크를 사용하지 않는다.
5. 배경 구조와 그림자를 의미론적으로 통일하지 않는다.
6. 실제 모델별 다중 참조 품질과 비용은 아직 측정하지 않았다.
7. 저사양 모바일의 참조 압축·정규화·ZIP 메모리 사용량은 실측하지 않았다.
8. 현재 환경에서는 브라우저 시각·접근성 E2E를 실행하지 못했다.
