# ViewGrid 운영 가이드

## 상태 확인

`GET /api/health`는 서비스 버전과 허용 공급자·모델 목록을 반환합니다. 외부 공급자를 호출하지 않으므로 앱 자체의 가용성 점검에만 사용합니다.

외부 공급자 상태는 사용자의 연결 확인 또는 각 공급자의 공식 상태 페이지로 별도 판단합니다.

## 모니터링 원칙

- 요청 URL, 상태 코드, 실행 시간과 ViewGrid 요청 ID만 사용합니다.
- API 키, FormData, 이미지 바이트, 프롬프트 전문은 기록하지 않습니다.
- 401·403 급증은 사용자 키 오류와 공급자 정책 변경을 구분합니다.
- 413 급증은 브라우저 전처리 또는 Vercel 페이로드 한도를 점검합니다.
- 429 급증은 Vercel WAF와 공급자 제한을 각각 확인합니다.
- 502·504 급증은 공급자 장애, 모델 폐기 또는 타임아웃을 확인합니다.

## 장애 대응

### 공급자 장애

1. `/api/health`가 정상인지 확인합니다.
2. 한 공급자만 실패하는지 비교합니다.
3. 오류 정규화가 키나 공급자 원문을 노출하지 않는지 확인합니다.
4. 사용자 매뉴얼의 우회 모델을 안내합니다.
5. 모델이 폐기됐다면 `docs/MODEL_UPDATE_GUIDE.md` 절차를 수행합니다.

### Vercel 413

1. 브라우저 전처리 Worker 폴백 여부를 확인합니다.
2. 원본 1.8MB, 가이드 1.05MB, 참조 900KB와 합산 3.75MB 제한을 확인합니다.
3. Vercel의 요청·응답 최대 페이로드보다 충분한 여유가 있는지 확인합니다.
4. 제한을 키우기보다 클라이언트 압축을 조정합니다.

### 과도한 호출

1. Firewall 로그에서 `/api/generate`를 확인합니다.
2. IP 기준 속도 제한과 Bot Management를 적용합니다.
3. 인스턴스 내 제한은 전역 보장이 아니므로 단독 방어로 사용하지 않습니다.
4. 필요하면 Preview 배포 보호와 Production 도메인만 허용합니다.

## 릴리스 절차

1. 버전과 `CHANGELOG.md`를 갱신합니다.
2. `npm ci`, `npm run check`, `npm run check:build`, `npm run verify:security`, `npm run audit:prod`를 실행합니다.
3. GitHub Actions의 quality, e2e, lighthouse를 확인합니다.
4. Preview 배포에서 스모크 테스트를 수행합니다.
5. Production으로 승격합니다.
6. `/api/health`, 보안 헤더, 실제 단일 생성과 다운로드를 재확인합니다.
7. 변경 후 24시간 동안 오류율과 Function 사용량을 관찰합니다.

## 롤백

Vercel의 이전 정상 배포로 즉시 Promote 또는 Rollback합니다. 모델 목록 변경만 문제라면 이전 `models.ts`와 공급자 어댑터를 복원한 패치를 우선합니다. 롤백 후 브라우저 캐시는 Service Worker 버전 변경으로 갱신합니다.
