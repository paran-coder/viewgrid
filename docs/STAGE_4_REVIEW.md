## 1. 단계 목표

Stage 4의 목표는 카메라 수치를 텍스트 프롬프트로만 전달하던 구조를 개선해, 브라우저에서 목표 구도를 나타내는 거친 가이드 이미지를 생성하고 원본과 함께 이미지 모델에 전달하는 것이었다.

구현 범위:

- Yaw, Pitch, Roll, FOV, Distance 기반 로컬 가이드
- 가이드 표시·숨김
- API 생성에서 가이드 사용·미사용 A/B 선택
- OpenAI와 Gemini의 다중 이미지 입력
- 가이드 생성 실패 시 원본 단독 폴백
- 가이드 설정 변경 후 오래된 결과 판별
- 요청 크기 제한과 가이드 메타데이터

## 2. 구현 결과

### 2.1 로컬 가이드 렌더러

`src/lib/camera-guide.ts`에 `perspective-strips-v1` 렌더러를 구현했다.

처리 순서:

```text
원본 이미지
→ 출력 프레임에 피사체 맞춤
→ FOV와 Distance로 피사체 점유율 계산
→ 세로 스트립 변형으로 Yaw 근사
→ 가로 스트립 변형으로 Pitch 근사
→ Roll 회전
→ 이동 오프셋 적용
→ 흐린 원본 배경과 합성
→ 제한 크기의 JPEG 가이드 생성
```

기본 제한:

- 최대 변: 1,024px
- 최대 파일 크기: 950KB
- 출력 형식: JPEG
- 렌더 방식: 브라우저 Canvas 2D
- 렌더러 버전: `perspective-strips-v1`

이 렌더러는 완전한 3D 복원이 아니라 카메라 방향, 화면 점유율과 원근을 모델에 시각적으로 전달하는 2D 휴리스틱이다.

### 2.2 편집 화면

새 `로컬 카메라 가이드` 패널을 추가했다.

- Image 1: 외형 기준 원본
- Image 2: 구도 기준 가이드
- 선택 카메라 변경 시 180ms 디바운스로 미리보기 갱신
- 가이드 API 사용 여부 스위치
- 미리보기 표시·숨김
- 가이드 파일 크기 표시
- 생성 중 설정 변경 방지

### 2.3 공급자 통합

OpenAI:

- `/v1/images/edits`
- `image[]` 첫 번째 항목에 원본
- `image[]` 두 번째 항목에 가이드

Gemini:

- Interactions API
- `input` 순서: 텍스트 → 원본 이미지 → 가이드 이미지

공통 프롬프트는 이미지 역할을 명시한다.

- Image 1은 형태·재질·색상·로고 기준
- Image 2는 카메라 방향·피사체 크기·화각·프레이밍 기준
- Image 2의 블러, 스트립 경계, 늘어짐과 빈 영역은 복사 금지

### 2.4 폴백과 상태 관리

- 가이드 기능을 사용자가 끄면 원본만 전송한다.
- 가이드 생성이 실패하면 전체 요청을 실패시키지 않고 원본만 사용한다.
- 결과 메타데이터에 `guideRequested`, `guideUsed`, `guideByteSize`, `guideVersion`, `guideWarning`을 기록한다.
- 결과 화면에 `GUIDE`, `FALLBACK`, `SOURCE` 상태를 표시한다.
- 가이드 사용 모드를 바꾸면 이전 결과를 `STALE`로 판정한다.
- 현재 가이드 모드와 일치하지 않는 결과는 3×3 완료 조건에서 제외한다.

### 2.5 요청 크기 정책

Vercel Function 요청 제한을 고려해 다음 안전 한도를 적용했다.

- 브라우저 전처리 원본: 최대 1.65MB
- 서버 원본 허용: 최대 1.8MB
- 카메라 가이드: 최대 950KB 생성, 서버 최대 1.05MB
- 원본+가이드 합계: 최대 2.85MB
- 전체 Function 요청: 최대 4.2MB 내부 제한
- 응답 이미지: 최대 4.2MB

## 3. 자체 점검에서 발견하고 수정한 문제

### 문제 1 — 가이드 모드 변경 후 이전 결과가 최신으로 보임

기존 `STALE` 판정은 카메라 숫자만 비교했다. 가이드 사용 결과를 만든 뒤 가이드를 끄거나 반대로 변경해도 결과가 현재 설정과 일치하는 것으로 표시됐다.

수정:

- 결과 메타데이터에 `guideRequested` 저장
- 현재 가이드 모드와 결과 스냅샷을 비교
- 불일치 시 `STALE` 표시
- 3×3 시트 완료 판정에서 제외

### 문제 2 — 원본과 가이드가 함께 전송되며 요청 크기 증가

Stage 3의 원본 최대 2.2MB를 유지하면 멀티파트 요청이 Vercel 한도에 가까워졌다.

수정:

- 원본 브라우저 한도를 1.65MB로 낮춤
- 가이드 최대 950KB 적용
- 서버에서 개별 크기와 합산 크기를 모두 검증

### 문제 3 — 가이드 실패가 전체 생성 큐를 중단할 가능성

가이드는 보조 입력이므로 로컬 Canvas 문제 때문에 유료 생성 전체가 중단되면 복구성이 낮다.

수정:

- 가이드 생성 오류를 카메라별 경고로 전환
- 프롬프트와 요청을 원본 단독 모드로 자동 폴백
- 결과에 `FALLBACK`과 원인을 기록

### 문제 4 — 가이드 이미지의 왜곡을 모델이 따라 할 위험

2D 스트립 기반 가이드는 블러와 늘어진 픽셀을 포함할 수 있다.

수정:

- 프롬프트에서 이미지별 역할을 명시
- 가이드의 블러·스트립 경계·늘어짐·빈 영역을 복사하지 말도록 명시
- 최종 이미지는 원본에서 깨끗하게 다시 구성하도록 지시

### 문제 5 — 미리보기 Blob URL 수명

카메라를 빠르게 바꿀 때 이전 가이드 URL이 남거나 이미 해제된 URL을 표시할 가능성이 있었다.

수정:

- 디바운스 작업 취소 처리
- 교체와 언마운트 시 Blob URL 해제
- 비동기 완료 전에 컴포넌트가 해제된 경우 즉시 URL 폐기

현재 환경의 `/usr/bin/chromium`은 조직 정책으로 `127.0.0.1`을 차단해 모든 테스트가 `ERR_BLOCKED_BY_ADMINISTRATOR`로 중단됐다. Playwright 전용 Chromium 다운로드도 실행 환경 DNS 제한으로 완료되지 않았다. 애플리케이션 결함이 아니라 실행 환경의 브라우저 정책이며, GitHub Actions에서는 전용 Chromium을 설치해 실행하도록 유지한다.

## 5. 알려진 한계

1. 가이드는 깊이 추정이나 3D 메시가 아닌 2D 스트립 원근 변형이다.
2. 인물, 투명체, 반사체, 복잡한 배경에서는 구도 힌트가 부정확할 수 있다.
3. 큰 Yaw와 Pitch에서는 가이드 자체의 늘어짐이 커진다.
4. 생성 모델이 가이드 각도를 수학적으로 정확히 따르는 것은 아니다.
5. 카메라별 가이드를 순차 생성하므로 저사양 모바일에서 CPU 사용량이 증가할 수 있다.
6. 실제 공급자 A/B 데이터가 없어 기본 가이드 강도는 휴리스틱 기준이다.
