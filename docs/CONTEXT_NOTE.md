## 2. 제품 정의

### 핵심 문장

> 프롬프트로 카메라 구도를 설명하는 대신, 이미지 위에 가상 카메라를 직접 배치해 원하는 각도와 화각의 멀티앵글 시트를 생성한다.

### 제품 유형

- 브라우저 기반 멀티앵글 이미지 생성 도구
- GitHub 공개 저장소로 관리
- Vercel에 배포
- 사용자가 자신의 OpenAI 또는 Google Gemini API 키를 입력
- 별도의 전용 GPU 서버, 회원가입, 데이터베이스, 영구 이미지 저장소는 운영하지 않음

### 핵심 출력

- 개별 카메라 결과 이미지
- 3×3 콘택트 시트 PNG
- 개별 결과 ZIP
- 카메라 파라미터가 포함된 프로젝트 설정 JSON

## 3. 핵심 차별점

일반적인 이미지 생성 채팅과 달리 다음을 구조화한다.

- 9개 가상 카메라 배치
- 좌우 각도(Yaw)
- 상하 각도(Pitch)
- 카메라 기울기(Roll)
- 화각(FOV)
- 거리 및 피사체 점유율
- 카메라 프리셋
- 일괄 생성 순서
- 3×3 시트 자동 조립
- 개별 셀 재생성

중요한 한계:

- 생성 모델이 실제 3D 렌더러처럼 입력 각도를 수학적으로 정확히 따르는 것은 아니다.
- 카메라 파라미터는 모델에 목표 시점을 더 명확하게 전달하기 위한 구조화된 제어 정보다.
- 큰 각도에서는 원본에 없는 영역을 모델이 추론해 생성하므로 제품 구조가 달라질 수 있다.

## 4. 확정된 제품 범위

### 기본 대상

- 배경과 피사체가 비교적 명확히 분리된 단일 제품 이미지
- 정면 또는 약한 사선 이미지
- 피사체 전체가 잘리지 않은 이미지

### 카메라 범위

| 파라미터 |  안정적 |    주의 |   실험적 | 하드 제한 |
| -------- | ------: | ------: | -------: | --------: |
| Yaw      |    ±25° |    ±45° |     ±60° |      ±60° |
| Pitch    |    ±10° |    ±20° |     ±30° |      ±30° |
| Roll     |    ±10° |    ±15° |     ±20° |      ±20° |
| FOV      | 35°–65° | 25°–80° | 20°–100° |  20°–100° |

초기 기본 프리셋:

- Yaw: `-25° / 0° / +25°`
- Pitch: `+10° / 0° / -10°`
- FOV: `50°`
- 중앙 셀: 원본 또는 0° 재생성 결과

### MVP에서 제외

- 완전한 360도 복원
- 정확한 뒷면 생성 보장
- 로그인 및 계정 시스템
- 서버 DB
- 서버 이미지 보관
- 결제 시스템
- 협업 및 공유 링크
- 모바일에서 고성능 로컬 3D 추론

## 5. 기술 구조

### 배포 구조

```text
GitHub Repository
       ↓
Vercel Deployment
       ↓
Next.js Client UI
├─ 이미지 업로드 및 전처리
├─ 카메라 편집
├─ 로컬 가이드 미리보기
├─ 결과 상태 관리
├─ 3×3 시트 생성
└─ PNG / ZIP 다운로드
       ↓
Vercel Route Handler / Function
├─ 입력 검증
├─ 사용자 API 키로 공급자 호출
├─ 응답 정규화
└─ 결과 반환 후 폐기
       ↓
OpenAI 또는 Google Gemini 이미지 API
```

### 현재 기술 스택

- Next.js 16 App Router
- React 19
- TypeScript strict
- Tailwind CSS 4
- Zustand 5
- Canvas API
- Vitest와 React Testing Library
- Playwright와 axe-core
- ESLint 9
- Prettier 3
- Vercel Functions

Stage 3에서 3×3 합성과 ZIP을 위해 Canvas API와 JSZip 사용 범위를 확장했고, Stage 4에서 Canvas 2D 기반 로컬 카메라 가이드 렌더러를 추가했다.

### 공급자 어댑터

실제 구현은 공급자별 연결 확인 함수와 생성 함수로 분리했다. 공통 UI는 `ProviderId`, `ImageModelId`, `ApiSettings`, `GeneratedImageMetadata` 타입만 사용한다.

지원 모델:

- OpenAI: `gpt-image-2`
- Google: `gemini-3.1-flash-image`
- Google 고품질 옵션: `gemini-3-pro-image`

모델 ID와 설명은 `src/lib/providers/models.ts`에서 관리한다. 공급자 요청 형식은 `openai.ts`, `gemini.ts`에 격리한다.

## 6. API 키 및 개인정보 원칙

### 원칙

- 사용자가 입력한 API 키는 브라우저 영구 저장소에 저장하지 않는다.
- 기본 상태는 React 메모리에서만 유지한다.
- 선택적으로 “현재 탭이 열려 있는 동안 유지”만 허용한다.
- API 키를 로그, 분석 도구, 오류 추적 도구에 전달하지 않는다.
- API 응답에 `Cache-Control: no-store`를 적용한다.
- 원본 및 결과 이미지를 DB나 서버 스토리지에 저장하지 않는다.
- 생성 결과는 브라우저에서 조립하고 내려받는다.

### Vercel 제약 대응

Vercel Function 요청과 응답은 각각 최대 4.5MB이므로 현재 다음 안전 한도를 적용한다.

- 브라우저에서 원본 최대 변을 1,536px로 축소
- 전처리 원본을 최대 1.65MB JPEG로 반복 압축
- Route Handler 원본 상한 1.8MB
- 로컬 가이드 최대 변 1,024px, 목표 상한 950KB JPEG
- Route Handler 가이드 상한 1.05MB
- 원본+가이드 합산 상한 2.85MB
- 전체 요청 안전 상한 4.2MB
- OpenAI 출력은 WebP 품질 85로 요청
- Gemini 출력은 1K JPEG로 요청
- 반환 이미지가 4.2MB를 초과하면 정규화된 오류 반환
- 9개를 한 요청에 묶지 않고 제한된 개별 큐로 처리
- API 키와 이미지 내용을 로그에 출력하지 않음

## 7. 생성 흐름

### Stage 2 기본 생성

```text
원본 이미지
→ 카메라 파라미터 선택
→ 카메라 설명 프롬프트 생성
→ 공급자 API 호출
→ 단일 결과 반환
```

### Stage 4 하이브리드 생성

```text
원본 이미지
→ 브라우저 Canvas에서 카메라별 목표 구도 가이드 생성
→ Image 1 원본 + Image 2 가이드 + 구조화 프롬프트
→ 공급자별 다중 이미지 API 호출
→ 가이드의 왜곡을 복사하지 않고 목표 구도를 보완한 결과 반환
```

가이드의 역할은 실제 3D 복원이 아니라 모델에 다음 정보를 시각적으로 전달하는 것이다.

- Yaw와 Pitch 방향
- FOV와 Distance에 따른 피사체 점유율
- Roll과 프레이밍
- 카메라별 목표 구도

가이드 생성에 실패하면 원본 단독 요청으로 폴백하고 결과 메타데이터에 원인을 기록한다.

### Stage 5 일관성 생성 순서

1. 중앙 C5
2. 왼쪽 중앙 C4
3. 오른쪽 중앙 C6
4. C1은 원본 + C4 + C1 가이드 참조
5. C2는 원본 + C5 + C2 가이드 참조
6. C3은 원본 + C6 + C3 가이드 참조
7. C7은 원본 + C4 + C7 가이드 참조
8. C8은 원본 + C5 + C8 가이드 참조
9. C9는 원본 + C6 + C9 가이드 참조

공급자가 허용하는 참조 이미지 수에 따라 우선순위를 조정한다.

## 8. UX 원칙

첨부된 UI 폴리시 지침을 프로젝트 UI 작업의 운영 원칙으로 사용한다.

1. 구조를 스타일보다 먼저 확정한다.
2. 장식보다 명확성을 우선한다.
3. 일회성 효과보다 재사용 가능한 컴포넌트를 우선한다.
4. 모든 인터랙션은 사용자의 카메라 제어 이해를 도와야 한다.
5. 모션은 상태 변화와 공간 관계를 설명할 때만 사용한다.
6. 블러와 그림자는 계층을 만드는 최소 범위에서 사용한다.
7. 실제 구현 가능성과 성능을 디자인 단계부터 검토한다.
8. 키보드 조작, 포커스 표시, 색상 외 상태 표시를 포함한다.
9. `prefers-reduced-motion`을 지원한다.

### 핵심 화면 우선순위

1. 이미지 업로드
2. 카메라 선택 및 조절
3. 품질 범위 표시
4. 생성 모델과 API 키 설정
5. 생성 진행 상태
6. 결과 시트 검토
7. 다운로드

## 10. 상태 및 인터랙션 규칙

### 품질 상태

- 안정적: 아이콘 + 텍스트 + 색상
- 주의: 아이콘 + 텍스트 + 색상
- 실험적: 아이콘 + 텍스트 + 색상

색상만으로 상태를 구분하지 않는다.

### 모션

- 패널 전환: 160–240ms
- 카메라 선택 강조: 120–180ms
- 결과 셀 등장: 240–400ms, 짧은 순차 지연
- 생성 중 진행 상태는 레이아웃 이동 없이 표시
- 무한 장식 애니메이션 금지

### 그림자 및 블러

- 기본 카드는 배경색과 1px 경계선으로 구분
- 모달과 팝오버에만 제한된 그림자 사용
- 상단 고정 바에는 필요할 때만 약한 배경 블러 적용
- 이미지 편집 캔버스 위의 텍스트는 블러보다 불투명 패널을 우선

## 13. 결정 기록

### 2026-07-19 — 초기 결정

- 제품은 GitHub에 공개하고 Vercel로 배포한다.
- 사용자가 자신의 OpenAI 또는 Google API 키를 사용한다.
- 전용 GPU 서버는 운영하지 않는다.
- Vercel Function은 API 키와 요청을 중계하지만 저장하지 않는다.
- 360도와 뒷면 복원을 제품 약속에서 제외한다.
- 카메라 파라미터 기반 멀티앵글 생성에 집중한다.

## 20. Stage 5 구현 기록

### 목표

Stage 4의 원본+가이드 생성 흐름에 **생성 계보, 인접 결과 참조, 출력 정규화, 설정 재사용**을 추가해 9개 결과가 하나의 제품 촬영 세트처럼 보일 가능성을 높인다.

### 생성 계획

```text
C5 → C4 → C6 → C2 → C8 → C1 → C3 → C7 → C9
```

참조 정책:

| 카메라 | 우선 참조 | 이유                                     |
| ------ | --------- | ---------------------------------------- |
| C5     | 없음      | 정면·중앙 기준 이미지                    |
| C4, C6 | C5        | 좌우 중앙 뷰가 중앙 외형을 공유          |
| C2, C8 | C5        | 중앙 열의 상하 뷰가 중앙 외형을 공유     |
| C1, C7 | C4        | 왼쪽 열의 상하 뷰가 왼쪽 기준을 공유     |
| C3, C9 | C6        | 오른쪽 열의 상하 뷰가 오른쪽 기준을 공유 |

- 비활성 카메라는 계획에서 제외한다.
- 재시도 시 현재 설정과 일치하는 완료 결과만 참조 후보로 사용한다.
- 선행 결과가 실패·삭제·STALE이면 참조 없이 원본+가이드로 폴백한다.
- 참조 파일은 브라우저에서 최대 1,024px·820KB WebP로 압축한다.
- Route Handler는 원본·가이드·참조 합산 3.75MB 안전 한도를 적용한다.

### 생성 입력 우선순위

```text
Image 1 원본
→ Image 2 로컬 가이드(선택)
→ Image 3 인접 완성 결과(선택)
```

프롬프트는 다음 우선순위를 명시한다.

1. 원본 외형·비율·색상·재질·로고가 최우선이다.
2. 가이드는 목표 카메라 방향과 프레이밍만 제공한다.
3. 인접 결과는 재질·구조·조명 일관성의 보조 기준이다.
4. 참조 결과와 원본이 충돌하면 원본을 따른다.
5. 새로운 문자·로고·부품·장식을 추가하지 않는다.

### 출력 정규화

- 생성 결과를 1,024×1,024 WebP로 통일한다.
- 원본 또는 실제 사용된 인접 결과의 샘플 평균색을 기준으로 제한된 RGB 게인을 계산한다.
- 채널 게인은 `0.84–1.16`, 휘도 게인은 `0.90–1.10` 범위로 제한한다.
- 과도한 보정을 막고 원본 제품 색상 변화 가능성을 낮춘다.
- Canvas 픽셀 접근이 불가능하면 색상 게인 없이 크기만 정규화한다.
- 결과 메타데이터에 정규화 버전, 크기, 게인, 기준 카메라와 경고를 기록한다.

이 정규화는 제품 검출·마스크 합성·의미론적 배경 교체가 아니다. 시트에서 결과 크기와 전역 색조 차이를 줄이는 보수적인 후처리다.

### 프리셋과 세션 복구

- JSON 스키마: `viewgrid-camera-preset` v1
- 저장 항목: 카메라 9개, 선택 카메라, 가이드 사용 여부, 참조 사용 여부, 정규화 사용 여부
- 불러오기 시 카메라 수·ID·범위와 데이터 타입 검증
- 범위를 벗어난 수치는 현재 카메라 하드 제한으로 보정
- `sessionStorage`에 동일한 설정만 탭 단위로 임시 보관
- API 키, 원본 이미지, 결과 이미지, Object URL과 공급자 응답은 저장하지 않음

### STALE 판정 확장

결과는 다음 조건 중 하나가 달라지면 현재 3×3 완료 조건에서 제외한다.

- 카메라 스냅샷
- 공급자와 모델
- 가이드 요청 모드
- 인접 참조 사용 모드
- 출력 정규화 사용 모드

### UI 상태

- 멀티뷰 일관성 패널에서 참조와 정규화를 독립적으로 켜고 끈다.
- 현재 생성 순서를 항상 표시한다.
- 생성 중 사용하려는 참조 카메라를 진행 상태에 표시한다.
- 결과 셀에 `REF C#` 또는 `NO REF`, `NORM` 또는 `RAW` 배지를 표시한다.

### 점검 중 발견하고 보완한 사항

1. 기존 C1→C9 순서는 상하·모서리 결과가 기준 이미지 없이 먼저 생성될 수 있어 중앙 기준 순서로 교체했다.
2. 현재 설정과 다른 공급자·모델·가이드·참조·정규화 결과가 시트에 섞이지 않도록 결과 컨텍스트 비교를 확장했다.
3. 인접 참조가 너무 커 Vercel 요청 한도를 넘지 않도록 별도 압축과 합산 크기 검증을 추가했다.
4. 참조 파일 생성 실패가 유료 큐 전체를 중단하지 않도록 참조 없는 폴백을 추가했다.
5. 같은 결과가 자신을 참조하거나 오래된 결과가 참조되는 것을 방지했다.
6. 전역 색상 정규화가 과도하게 제품색을 바꾸지 않도록 게인 범위를 제한했다.
7. 세션 복구 데이터에서 API 키와 이미지가 구조적으로 포함되지 않도록 프리셋 스키마를 분리했다.
8. 카메라 수동 조정 또는 일관성 옵션 변경 시 기존 결과가 `STALE`로 갱신되도록 했다.

## 20. Stage 6 구현 기록 — 공개 운영 완성

### 보안 기준선

- Next.js 전역 응답에 CSP, HSTS, Referrer Policy, Permissions Policy, `X-Frame-Options`, `X-Content-Type-Options`, COOP와 CORP를 적용한다.
- CSP는 기본 출처를 동일 출처로 제한하고, Pretendard CDN 스타일·폰트와 브라우저 Blob 이미지만 허용한다.
- `/api/connection`과 `/api/generate`는 브라우저 출처, 콘텐츠 형식, 본문 크기, 키 길이, 공급자·모델·MIME·개별·합산 이미지 크기를 검증한다.
- API 요청마다 무작위 ViewGrid 요청 ID를 부여하고 응답은 `no-store`로 반환한다.
- Production에서는 IP 기준 인스턴스 단위 속도 제한을 적용하되, 전역 방어는 Vercel Firewall 규칙을 운영 필수 항목으로 둔다.
- Service Worker는 `/api/`, 사용자 업로드와 생성 Blob을 캐시하지 않는다.

### PWA와 성능

- Metadata Manifest와 192·512·Maskable·Apple 아이콘을 제공한다.
- 정적 앱 셸과 Next 정적 청크만 캐시하는 Service Worker를 제공한다.
- 오프라인 상태를 사용자에게 알리고 실제 API 생성은 온라인에서만 수행한다.
- 원본과 인접 참조 압축은 OffscreenCanvas Worker를 우선 사용한다.
- Worker 미지원, 이미지 디코딩 실패, WebP 인코딩 실패 또는 타임아웃 시 메인 스레드 Canvas 경로로 폴백한다.
- 결과 이미지는 비동기 디코딩과 아래 행 지연 로딩을 사용한다.
- Canvas는 사용 후 1×1로 축소하고 Worker, ImageBitmap과 Blob URL을 종료·해제한다.
- 프로덕션 JavaScript 청크 gzip 합계와 최대 청크 예산을 자동 검사한다.

### 접근성

- 본문 바로가기 링크를 제공한다.
- 슬라이더에 레이블, 도움말 연결과 `aria-valuetext`를 제공한다.
- 생성 영역에 `aria-busy`, 진행률에 `progressbar` 의미를 적용한다.
- API 설정과 결과 확대 모달에 Escape, Tab 순환과 포커스 복귀를 적용한다.
- 터치 입력에서는 버튼과 슬라이더 조작 영역을 최소 44px 수준으로 확대한다.
- 상태는 색상 외에 텍스트·아이콘·배지와 live region으로 전달한다.
- 축소 모션 사용자의 애니메이션과 전환을 최소화한다.

Next.js 16.2.10 내부의 PostCSS가 8.5.10 미만으로 설치되며 중간 등급 취약점이 보고됐다. Next.js 메이저 다운그레이드 대신 npm `overrides`로 해당 하위 의존성을 8.5.10에 고정했고, 프로덕션 감사 결과 알려진 취약점 0건을 확인한다.
